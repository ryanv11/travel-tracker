# UX Audit — Travel Tracker
**Date:** 2026-03-19
**Author:** UX (UI/UX Designer)
**Brief:** BRIEF-UX-01
**Status:** Final

---

## Methodology

Every component and page in `/workspace/src/frontend/` was read in full before any finding was written. Styling is 100% inline React `CSSProperties` — no external CSS files, no Tailwind, no component library. All values are hardcoded at the call site throughout the codebase.

---

## Global / Cross-Cutting Findings

### G-01 — Inline styles everywhere — no design system
**Severity: MAJOR**
Every colour, spacing value, font size, and border radius is hardcoded as a literal at the point of use. The same value (e.g. `#2563EB`, `#D1D5DB`, `'6px'`) is repeated across dozens of files with no shared token. This means:
- Any global change (e.g. primary colour) requires find-and-replace across ~35 files
- Values drift between files (e.g. `borderRadius: '5px'` in some places, `'6px'` in others, `'8px'` in others)
- There is no single source of truth for the visual language

This is the foundational defect that all other issues sit on top of. The migration to Tailwind + shadcn/ui resolves this structurally.

### G-02 — No focus styles on any interactive element
**Severity: MAJOR**
No component applies a visible `:focus` or `focus-visible` outline. The browser default focus ring is suppressed by the global reset (`margin: 0; padding: 0`). Every button, input, select, and link is currently keyboard-inaccessible in practice. This is an accessibility violation (WCAG 2.1 SC 2.4.7 — Level AA).

### G-03 — No aria-label on most buttons
**Severity: MAJOR**
Most action buttons render only a text label with no `aria-label` or `aria-describedby`. Icon-only or ambiguous buttons (e.g. the "×" clear-filter button, the emoji-labelled delete/edit buttons on cards) are opaque to screen readers. The geocoding indicator dismiss button has `title` but no `aria-label`.

### G-04 — Emoji used for semantic icons throughout
**Severity: MINOR**
Item type icons (✈️ 🏨 🍽️ etc.), the nav bar logo (✈️), the lock indicator (🔒), the photo album link (📷), and the geocoding pending indicator (☁) are rendered as plain Unicode emoji. Emoji are not scalable vectors, do not respect system colour inversion, are inconsistent across OS/browser, and cannot be reliably styled. They read poorly for screen readers and render at low quality at small sizes. Lucide icons (the shadcn/ui default, already recommended) should replace all of these.

### G-05 — No toast/notification system
**Severity: MAJOR**
There is no global feedback mechanism for successful mutations. After creating a trip, adding a place, editing an item, or any write operation, the app provides no confirmation that the action succeeded — the modal just closes. The only feedback for the carry-forward flow is a green success box inside the modal before it auto-closes. Every other success state is invisible. This is a feedback loop failure. A shadcn/ui Toast component is required.

### G-06 — Inconsistent button border radius
**Severity: MINOR**
Buttons use `borderRadius: '5px'`, `'6px'`, `'8px'`, and `'9999px'` (pill) across the codebase with no discernible rule. The admin tab buttons use `borderRadius: '0'`. This fragmentation undermines visual consistency.

### G-07 — Loading spinner injects a `<style>` keyframe on every render
**Severity: MINOR**
`LoadingSpinner.tsx` inserts `@keyframes spin` into the document via a `<style>` tag inside the component render. This causes repeated style injection on re-renders. In the Tailwind migration this will be replaced with `animate-spin` from Tailwind's animation utilities, which resolves the issue structurally.

### G-08 — Trips page and Trip Detail page have double-padding
**Severity: MINOR**
`TripsPage` applies `padding: '24px 16px'` to its container and passes that down to `TripList`, which then also applies `padding: '24px'` to its own root div. This results in `48px` of top/left padding on the trips page — significantly more than intended. The same pattern occurs in `TripDetailPage` (outer `24px 16px`) → `TripDetail` (inner `24px`). The content is not as wide as the `maxWidth: '900px'` container visually suggests.

### G-09 — Country code shown raw in place headers
**Severity: MINOR**
`PlaceSection` renders the city's country code (e.g. `AU`, `US`, `GB`) directly as `{place.city.country_code}`. Country name is available from context — this should be the full country name or at minimum a flag emoji + code. As raw code it is cryptic for the user.

## Screen-by-Screen Audit

---

### Screen 1: Map Page (`/map`)

#### 1.1 Layout & Density
The map takes the full viewport below the nav bar. This is correct — the map is the centrepiece and should have all available space. No dead space.

**Finding M-01 — Loading state overlaps map controls (MINOR)**
When `shadingLoading` is true, the spinner is positioned `top: 12, left: 12` absolutely. MapLibre's own attribution and zoom controls also appear at bottom-right. The spinner overlaps with any MapLibre UI elements in the top-left corner. It should be a pill badge positioned in the top-right, not covering the default MapLibre control area, or a thin progress bar across the top.

**Finding M-02 — No legend for map shading states (MAJOR)**
The map shows countries in up to 6 different colours (configurable by the user). There is no legend anywhere on the map page to explain what each colour means. A user who hasn't memorised their own shading configuration sees coloured countries with no explanation. A collapsible legend overlay — small, bottom-left, non-competing — is required.

**Finding M-03 — No empty state message for zero trips (MINOR)**
When no trips are logged, the map renders unshaded. There is no onboarding message, no call-to-action, and no indication that the user should go to /trips to add their first trip. The map just looks empty with no affordance.

**Finding M-04 — City marker icons rely on MapTiler's 'marker' sprite (MINOR)**
`CityMarkers` specifies `'icon-image': 'marker'` — this relies on the MapTiler tile style including a sprite named `marker`. If the user switches tile styles or the MapTiler street style changes its sprite names, city markers will silently disappear. This is a fragility risk. A custom sprite or fallback should be used. This is on the architectural boundary — flagged to COO.

#### 1.2 Visual Hierarchy
The map dominates as intended. The nav bar sits above it cleanly. Hierarchy is appropriate.

#### 1.3 Consistency
The map page is the most visually coherent page because it has the fewest UI elements. Consistent with the rest of the app's neutral background.

#### 1.4 Feedback & States
Loading state exists (spinner). Error state exists (red ErrorMessage box). Empty state is absent (see M-03).

#### 1.5 The Map
Map is given correct visual priority. The `height: '100%'` chain is properly maintained from `#root` through `App` → `main` → `MapPage` → `MapView`. This is the structural decision that makes the map work.

### Screen 2: Trips List (`/trips`)

#### 2.1 Layout & Density
**Finding T-01 — Page width underutilised (MAJOR)**
The trips list is constrained to `maxWidth: '960px'` (on `TripsPage`) with an inner `maxWidth: '900px'` (on `TripList`), and both apply their own `padding`. On a typical 1440px or 1920px desktop display, the list floats as a narrow column with large amounts of dead white space on both sides. This is a desktop data application — the list should use more of the horizontal space, or split into a sidebar + detail panel rather than a centred narrow column.

**Finding T-02 — Filter controls consume disproportionate vertical space (MINOR)**
There are two filter rows stacked: one for search + sort, one for status + category + activity. This is 2× the vertical height of the first trip card. The filter controls should be in a single horizontal row, or collapsed behind a "Filters" button when not active.

**Finding T-03 — Trip cards are too sparse (MAJOR)**
Each `TripCard` uses `padding: '16px'` top and bottom, with `marginBottom: '12px'` between cards. For a list that may contain 20+ trips, this means significant scrolling. The card should be tighter — 10–12px padding, 8px gap. The internal content of the card (name, dates, companions, categories) is vertically stacked with generous `marginTop` values that could be reduced.

#### 2.2 Visual Hierarchy
**Finding T-04 — "Edit" and status badge compete for equal visual weight in card header (MINOR)**
In the card header, the trip name (h3, 16px, weight 600) shares the row with a status badge and an Edit button. The Edit button is styled identically to the status badge in terms of visual weight. The Edit button should be lower weight — a ghost/outline style or a small icon button. The trip name should dominate the header row.

**Finding T-05 — Date format inconsistency between TripCard and TripDetail (MINOR)**
`TripCard` uses `formatDate()` → "1 Jun 2026". `TripDetail` renders `{trip.start_date}` directly → "2026-06-01" (ISO string). The same data is presented in two different formats on adjacent views. Date formatting must be consistent.

#### 2.3 Consistency
**Finding T-06 — Map filter badge uses a different visual language from StatusBadge (MINOR)**
The map filter indicator (`background: '#EFF6FF', border: '1px solid #BFDBFE'`) looks like a custom one-off. It should use the same badge design system as `StatusBadge`.

#### 2.4 Feedback & States
Loading state exists. Error state exists. Empty state exists (plain text message). The empty state message is adequate but unstyled — it should be a designed empty state with an action button to create the first trip.

**Finding T-07 — No success feedback on Trip Create (MAJOR)**
Creating a trip closes the modal silently. No toast, no highlight, no indication which card is the new one. Per G-05, this is a systemic gap.

### Screen 3: Trip Detail (`/trips/:id`)

#### 3.1 Layout & Density
**Finding TD-01 — Dates rendered as raw ISO strings (MAJOR)**
`TripDetail` renders `{trip.start_date} – {trip.end_date}` directly from the API response (format: `YYYY-MM-DD`). These are machine-readable strings, not human-readable dates. All date display must use a formatting function. This is also an inconsistency with TripCard (see T-05).

**Finding TD-02 — Status transition buttons have no visual distinction from each other (MAJOR)**
The status transition buttons (e.g. "Mark Active", "Move to Review", "Lock Trip") are all styled identically — `border: '1px solid #D1D5DB'`, `background: '#fff'` — except "Lock Trip" which gets a yellow tint (`#FEF3C7`). These are semantically different actions: some are neutral progressions, one is destructive/final (Lock). They should follow a clear visual hierarchy: primary action → filled button, secondary → outline, destructive → red tint or warning colour.

**Finding TD-03 — Locked banner is low contrast and visually neutral (MINOR)**
The locked banner (`background: '#F3F4F6', border: '1px solid #D1D5DB', color: '#374151'`) looks like a disabled element, not a deliberate read-only status indicator. It needs stronger visual treatment — a distinct background and icon — to communicate that the trip is permanently in a read-only state.

**Finding TD-04 — "Add Place (City)" button is dashed-border placeholder style (MINOR)**
The add-place button uses `border: '2px dashed #D1D5DB'` — a "dropzone" visual metaphor. This is inconsistent with the filled primary "+ Add Item" button inside each PlaceSection. The add-place action should be a proper outlined or ghost button, consistent with the rest of the action hierarchy.

**Finding TD-05 — Category tags duplicated in TripCard and TripDetail with different implementations (MINOR)**
Both `TripCard` and `TripDetail` render category chips with inline `tagStyle` defined locally. The two implementations are nearly identical but separately defined. This is a design system gap — `CategoryChip` should be a shared component.

#### 3.2 Visual Hierarchy
The trip name is clearly dominant (h1, 24px, weight 700). Status badge sits correctly alongside. Companions and categories read as secondary. PlaceSection cards visually group content appropriately.

**Finding TD-06 — PlaceSection header uses country_code not country name (see G-09)**

#### 3.3 Consistency
**Finding TD-07 — "Add Item" button inside PlaceSection is filled primary blue; all other secondary actions are outlined (MINOR)**
The "+ Add Item" button within each PlaceSection header is styled as a primary filled button (`background: '#2563EB'`). This creates a hierarchy conflict — it appears more prominent than the "Edit" button in the trip header. Item addition is a secondary action within the context of a trip; it should be an outlined button, with only the top-level primary action (the trip CTA at the list level) using filled primary.

#### 3.4 Feedback & States
**Finding TD-08 — No loading state on status transition buttons during mutation (MINOR)**
While a status transition is pending (`updateStatus.isPending`), the buttons become `disabled` but show no loading indicator. The user sees no visual confirmation that the action is in progress. A spinner or label change ("Saving…") is required.

**Finding TD-09 — PlaceSection empty item state is adequate but not designed (POLISH)**
"No items yet. Add one with '+ Add Item'." is functional but plain text. Should be a small styled empty state.

### Screen 4: Item Form (Modal)

#### 4.1 Layout & Density
**Finding IF-01 — Type selection grid is oversized (MINOR)**
Step 1 of ItemForm shows 6 type buttons in a 3×2 grid, each with `padding: '14px'` and a 24px emoji icon. This takes up nearly the full modal height for a step that is just a type picker. The type selector should be more compact — a horizontal chip strip or a smaller 2-column grid.

**Finding IF-02 — Flight form has 6 date/time fields in a 2-column grid creating very narrow inputs (MINOR)**
The flight form places all 6 fields (airline, flight number, departure airport, arrival airport, departure datetime, arrival datetime) in a `gridTemplateColumns: '1fr 1fr'` layout. The datetime fields use `type="datetime-local"` which has a browser-native picker that is very cramped at half the modal width (roughly 220px). The airport code fields (short inputs) and the datetime fields (wide pickers) should not be in the same grid column width.

#### 4.2 Visual Hierarchy
The two-step flow (type selection → fields) is logical. Modal title "Add Item" / "Edit Item" is clear.

**Finding IF-03 — Status dropdown appears at the top of the form, before type-specific fields (MINOR)**
Status is a secondary attribute — the primary data is the type-specific content (property name, airline, etc.). Placing the Status dropdown first means the user must set status before entering the actual item content. Status should be near the bottom, adjacent to Notes, as a secondary field.

#### 4.3 Consistency
Field component `Field` is consistent internally. Good pattern.

#### 4.4 Feedback & States
**Finding IF-04 — No loading state indicator when modal first opens with existingItem (POLISH)**
Edit mode pre-populates from the passed `existingItem` prop — no async required, so no loading state needed here. This is fine.

### Screen 5: Add Place Flow (Modal)

#### 5.1 Layout & Density
Compact and appropriate for its purpose.

**Finding APF-01 — Search results list has no empty-state message when search returns 0 results (MINOR)**
When `debouncedQuery.length >= 2` but `searchResults.length === 0` (and not searching), the result list container renders with just the "+ Add new" row. There is no "No cities found matching X" message. The user may not understand why no results appeared.

**Finding APF-02 — "Searching…" and results can appear simultaneously (POLISH)**
When `searching` is true AND `debouncedQuery.length >= 2`, both the "Searching…" text and any previously cached results list would render. This is a minor visual flicker but should be handled with a clear sequence.

#### 5.2 Visual Hierarchy
Clear flow. "Add new" option at the bottom of results is well-positioned.

#### 5.3 Feedback & States
Loading state (searching) present. Error state present. The `mutationError` is shown in two places when `showNewCityForm` is false — once above the results area and once below the form (line 239). This should be rationalised.

### Screen 6: Post-Trip Review Panel

#### 6.1 Layout & Density
**Finding RP-01 — Review panel maxWidth: '800px' is inconsistent with Trip Detail maxWidth: '900px' (MINOR)**
The review panel constrains to 800px while trip detail uses 900px. These two views are reached from the same URL and should share the same layout container width.

**Finding RP-02 — "Mark all as Completed" button has no visual feedback during mutation (MINOR)**
The button becomes `disabled={updateItem.isPending}` but shows no loading indicator. This mutation can take several seconds if the trip has many items (each is a sequential PATCH call). The user sees a frozen button with no progress. This should have a spinner or progress indicator.

**Finding RP-03 — Sequential PATCH calls for bulk completion (ARCHITECTURAL FLAG — see COO section)**
`handleMarkAllCompleted` fires individual `await updateItem.mutateAsync()` calls sequentially for every `consider/confirmed` item. This is N sequential API round-trips. For a trip with 20 items this could be 20 sequential network calls. A bulk update endpoint would be correct. This is flagged to COO as an architectural concern.

#### 6.2 Visual Hierarchy
**Finding RP-04 — "Complete Review & Lock Trip" uses red (#DC2626) which signals destructive/error (MAJOR)**
The primary action on the review page — locking the trip — uses the same red as the delete confirmation button and the error state colour. Red conventionally signals danger/destruction, not completion. The lock action is a positive completion; it should use green (`#059669` / `#065F46`) or a distinct action colour. This is a significant semantic colour error.

#### 6.3 Consistency
The ReviewItemRow is well-structured. The inline status select + optional rating textarea is a clean pattern.

**Finding RP-05 — ReviewItemRow post-visit notes saved on blur, not on change (POLISH)**
This is intentional UX to avoid thrashing — blur is the correct pattern. No issue.

### Screen 7: Carry-Forward Modal

#### 7.1 Layout & Density
Appropriate modal size. The checkbox list is clean.

**Finding CF-01 — All items pre-selected by default — user must actively deselect (MINOR)**
`useState<Set<number>>(new Set(candidates.map((c) => c.id)))` initialises all candidates as selected. If the user has 10 carry-forward candidates and wants none, they must uncheck 10 boxes. The expected behaviour for a suggestion flow is unselected by default — the user actively opts in. This is a significant UX inversion.

#### 7.2 Visual Hierarchy
"Carry-forward suggestions" title is clear. The "From: [trip name]" source attribution is excellent — it gives the user context to make the decision.

#### 7.3 Feedback & States
Success state exists (green confirmation box + auto-close). Error state exists. Loading state via button label change.

### Screen 8: Admin Panel

#### 8.1 Layout & Density
**Finding AD-01 — Countries tab renders up to 250 items with no virtualisation (MINOR)**
`CountryTab` renders all 250 countries in a scrollable `maxHeight: '500px'` div. At `~38px` per row, all 250 countries render as DOM nodes simultaneously. This is a minor performance concern and the scroll UX is poor — the country list sits inside a page that itself scrolls. Consider a proper list with stable height and search-first interaction.

**Finding AD-02 — Admin panel is completely unstyled relative to the rest of the app (MINOR)**
`AdminPage` passes directly to `AdminPanel` with no page-level chrome. The admin panel has its own `padding: '24px 16px'` and title, but it lacks any visual differentiation from the trips pages. An admin section should have a distinct visual treatment (e.g. a top border in the admin colour, a subtle page background tint) to communicate "you are in a configuration area".

#### 8.2 Visual Hierarchy
Tab bar is clear. Active tab indicated by blue underline and bold weight. Content area is well-structured.

**Finding AD-03 — "Deactivate" (danger) and "Rename" (secondary) buttons are the same size and proximity (MINOR)**
In the list admin tabs, "Rename" and "Deactivate" appear next to each other with no visual separation. A destructive action (Deactivate) should not sit flush against a non-destructive action (Rename) without spacing or visual distinction. Risk of accidental deactivation.

#### 8.3 Consistency
Categories, Activities, and Companions tabs are implemented as near-identical components (see G-01 — inline styles duplicated three times). These are the same design pattern with different data sources — ideal candidates for a shared `AdminListTab` component or at minimum shared style tokens.

**Finding AD-04 — Map Shading tab: colour picker fires mutation on every change event (MINOR)**
`ShadingTab` uses `onChange` on `<input type="color">` to fire `updateColor.mutateAsync`. Browser colour pickers fire `onChange` continuously as the user drags the picker — this will fire dozens of PATCH requests as the user adjusts a colour. Should use `onBlur` or `onChangeComplete` equivalent (a stable selection event) instead of `onChange`.

#### 8.4 Feedback & States
Loading states exist in all admin tabs. Error states exist. No success feedback on save (see G-05).

## Architectural Concerns Flagged to COO

### ARCH-01 — Sequential PATCH calls in bulk review completion
`ReviewPanel.handleMarkAllCompleted()` fires individual API calls sequentially per item. For trips with many items this will be slow and fragile. A `POST /api/trips/:id/items/bulk-update` endpoint is the correct solution. This is a backend API gap, not a frontend bug.

### ARCH-02 — City marker icon sprite dependency on MapTiler
`CityMarkers` references `'icon-image': 'marker'` which assumes the MapTiler Streets v2 style includes this sprite. This is not guaranteed across style versions or if the user switches to an alternative tile provider. A custom SVG marker or a MapLibre-defined symbol should be used instead. Risk: silent disappearance of all city pins.

### ARCH-03 — ShadingTab fires mutations on colour picker drag
The `onChange` handler on `<input type="color">` fires during the drag motion of the OS colour picker, not on settled selection. This generates dozens of PATCH requests per colour adjustment. Should use `onBlur` or capture value on mouseup/pointerup. Potentially causes database churn and race conditions.
