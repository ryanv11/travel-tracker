## New Findings — File as GitHub Issues

### BUG-11 (Medium) — PATCH trip: end_date bypasses date validation when start_date omitted
**File:** `src/backend/validation/trips.schemas.ts`
`UpdateTripSchema` refine returns `true` when only one date field is present. A PATCH with only `end_date` can set it before the stored `start_date` without error.
**Repro:** `PATCH /api/trips/:id` with `{"end_date": "2020-01-01"}` on a trip with `start_date: "2026-06-01"` → 200 OK, corrupted date range.
**Fix direction:** Validate `end_date` against stored `start_date` in route handler, or require `start_date` whenever `end_date` is patched.

---

### BUG-12 (High) — Admin list endpoints return camelCase keys; all items render as "Inactive"
**File:** `src/backend/routes/admin.ts` — `createAdminListRouter()` factory
`res.json(rows)` emits raw Drizzle camelCase (`isActive`, `createdAt`). Frontend reads `item.is_active` → `undefined` → all items appear Inactive in admin UI.
**Affected endpoints:** `GET /api/admin/categories`, `GET /api/admin/activities`, `GET /api/admin/companions`
**Fix direction:** Add explicit snake_case serialization in the factory (same pattern as the countries endpoint).

---

### SEC-01 (High) — GET /api/cities/:id/items returns all users' completed items
**File:** `src/backend/routes/cities.ts`
Query has no `userId` filter — any authenticated user can retrieve completed items belonging to other users.
**Fix direction:** Add `eq(items.userId, userId)` to the where clause.

---

### SEC-02 (Medium) — Carry-forward source_item_ids lack ownership check
**File:** `src/backend/routes/places.ts` — POST `/:placeId/carry-forward`
`foundItems` query does not verify items belong to the current user. User A can carry forward user B's items.
**Fix direction:** Add `eq(items.userId, userId)` to the existence check.

---

### SEC-03 (Medium) — DELETE place activity has no trip/user ownership check
**File:** `src/backend/routes/places.ts` — DELETE `/:placeId/activities/:activityId`
Deletes by `placeId + activityId` only — no verification that the place belongs to the requesting user's trip.
**Fix direction:** Verify place ownership via `placeRepository.findById(userId, placeId)` before deleting.
