## What was decided

## Key decisions at a glance

**Schema:** Add `arrived_on TEXT` and `departed_on TEXT` (nullable, no default) to
`trip_places`. Naming chosen over `date_from/to` and `start_date/end_date` for
domain clarity. Migration is two `ALTER TABLE ADD COLUMN` statements — no table
recreation, safe with the patched drizzle-kit.

**New endpoint required:** `PATCH /api/trips/:tripId/places/:placeId` does not
exist yet. DP-05 requires editing existing place dates, so this endpoint must be
created as part of the feature.

**Ordering:** Client-side sort in the trip detail component. Nulls last (undated
places preserve insertion order per BRD). Stable sort — same arrival date keeps
insertion order.

**DP-04 precedence (three-source model):**
1. Explicit `arrived_on` / `departed_on` (highest priority)
2. Hotel check-in / check-out dates
3. Trip start / end dates (fallback)

**Partial dates:** Only `arrived_on` set (no `departed_on`) is valid and supported.
The sort uses `arrived_on`; display shows arrival-only. "Only `departed_on` set"
is permitted by schema; sort treats it as undated (null `arrived_on`).

**Out-of-range dates:** Backend warns but does not block. Warning returned in
response body; frontend should surface it.

**Effort:** ~10 hours total across backend + frontend.
