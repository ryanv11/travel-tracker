## Decisions made

### AD-07 — Map shading config (per-user)

**Decision: Add `user_id` FK to `map_shading_config`. Composite PK `(state_key, user_id)`.**

Rejected the sparse-override/global-defaults approach (Option B). Global defaults create
an ownership coupling where an owner's config changes retroactively affect all users who
haven't overridden that state — violating AD-07's isolation guarantee. Option A (6 rows
per user) is simpler, fully isolated, and scales trivially.

Lazy seeding: when a new user first accesses shading config and has 0 rows, the repository
seeds their 6 defaults from hardcoded application constants. No schema change needed for
seeding; no user-creation hook required.

### AD-08 — Companions (per-user)

**Decision: Add `user_id` FK to `companions`. UNIQUE changes from `(name)` to `(user_id, name)`.**

Cascade delete on `user_id` ensures companion rows are cleaned up when a user is deleted.

**Cross-user companion enforcement:** `trip_companions_map` cannot be DB-constrained to
prevent cross-user companion assignment (SQLite limitation). The backend must validate
companion ownership in `tripRepository.replaceAssociations` before inserting junction rows,
using a new `companionRepository.validateOwnership(userId, companionIds)` function.

### SE-03 scope change

AD-07/08 deliberately narrows SE-03. Companions and map shading config move from
owner-only to per-authenticated-user. This is the intended BRD v2.5 admin split model.

**Remaining SE-03 scope (still owner-only):**
- Categories (`/api/admin/categories`)
- Activities (`/api/admin/activities`)
- City creation (`POST /api/cities`)
- Country/region config (`/api/admin/countries`, `/api/admin/countries/:code/regions`)

### AD-09 — Unaffected

`trip_categories` and `activities` tables are unchanged. No `user_id` column. Still
global seeded defaults, still owner-gated for deactivation.


Both migrations use table recreation (SQLite `ALTER TABLE` limitation). Both include a
data migration step: `INSERT ... SELECT ... CROSS JOIN (SELECT id FROM users WHERE
is_owner = 1 LIMIT 1)` to assign existing global rows to the owner.

**Edge case:** If no owner user exists at migration time (dev fresh DB), existing global
rows are discarded (acceptable; lazy-seed handles first access).

## Key implementation guidance for Backend agent

1. **Shading service data isolation is a bug fix, not just a config change.** The current
   `getAllCountryShading`, `getCountryShading`, `getRegionShading`, `getCityShading` all
   aggregate trips across ALL users (no `WHERE trips.user_id = ?`). This is a SE-02
   violation. AD-07 fixes this — every shading aggregate query must filter by `userId`.

2. **Per-user config cache.** The current `_configCache` is a module-level singleton. After
   per-user config it must be a `Map<userId, ShadingConfigMap>`. `invalidateConfigCache()`
   must accept `userId` and invalidate only that user's cache entry.

3. **Companion route relocation.** Move companion endpoints from `/api/admin/companions`
   to `/api/companions`. The `adminRouter` router-level `requireOwner` guard must not be
   worked around with per-route overrides — extraction is cleaner.

4. **Migration data migration.** Drizzle's generated migration will produce the table
   recreations. The Backend agent must manually add the `INSERT ... SELECT ... CROSS JOIN`
   data migration statement into the generated SQL file before running `db:migrate`.

5. **Test updates.** Contract tests and unit tests that seed `companions` or `mapShadingConfig`
   rows globally must be updated to seed per-user rows (with a `user_id`). Tests for
   companion-trip associations must include the ownership validation path.

## Risks flagged

- **R1 (critical):** Shading trip-data isolation was absent — fix is mandatory with AD-07.
- **R2:** Per-user cache invalidation scope — must not be global after this change.
- **R3 (acceptable):** Migration data loss on no-owner dev DB — documented, mitigated by lazy-seed.
- **R4:** Cross-user companion enforcement is application-only — must be test-covered.
