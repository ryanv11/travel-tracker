DEVIATIONS FROM SPEC
================================================================================

1. CORS configuration: Original spec said "allow all origins for beta."
   Security addendum overrode this with the allowlist pattern. Implemented
   the allowlist (ALLOWED_ORIGINS env var). Beta clients must be on the list.

2. Helmet frameguard: Default Helmet v8 uses X-Frame-Options: SAMEORIGIN.
   We applied frameguard: { action: 'deny' } to set DENY as required by SEC-01.

3. Zod v4 compatibility: Zod v4 changed error.errors → error.issues.
   Applied fix in validate.ts. Documented in code comments.

4. Express v5 compatibility: req.query is read-only in Express v5.
   Used Object.defineProperty to override after validation.
   Documented in validate.ts header comment.

5. data/regions.json covers only US (51), AU (8), CA (13) = 72 entries
   as specified in Correction 2. All other region-tier countries (DE, IN,
   JP, etc.) can have regions added via the admin panel.

1. CARRY-FORWARD EXECUTION ENDPOINT (IT-07 partial):
   The spec defines GET /api/cities/:id/carry-forward (returns candidate items)
   and describes executeCarryForward() as a service function. However, there is
   no POST endpoint that FRONTEND can call to actually execute the carry-forward
   and create the new items on a target trip.

   BACKEND has implemented the service function (executeCarryForward in
   items.service.ts). FRONTEND would need an endpoint to invoke it.

   Proposed: POST /api/trips/:tripId/places/:placeId/carry-forward
     Body: { source_item_ids: number[] }

   Currently not implemented. FRONTEND should NOT attempt to call this endpoint.
   Awaiting COO direction on whether to implement now or defer to Phase 2.

2. CITY PATCH ENDPOINT:
   The spec does not define PATCH /api/cities/:id. The corrections mention
   a city can have region_id "set later via PATCH." This endpoint does not exist.
   Awaiting COO direction.
