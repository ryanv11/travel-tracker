KEY DECISIONS MADE THIS THREAD
================================================================================

  1. BACKEND Phase 1 ACCEPTED
     Rationale: All 26 ACs verified. Code spot-check confirms correctness on
     all critical paths. Deviations (CORS override, Helmet frameguard, Zod v4
     issues adaptation) were appropriate and correctly documented.

  2. Carry-forward POST endpoint URL design
     Chose: POST /api/trips/:tripId/places/:placeId/carry-forward
     Rationale: Follows the nested resource pattern (place is the destination
     context). Body carries source_item_ids. Keeps endpoint semantics clear
     (this is about what you are carrying INTO a place, not FROM a city).

  3. City POST rate limit set to 20/min (not 10 as BACKEND suggested)
     Rationale: A user entering historical trip data may create many cities
     in rapid succession. 10/min is too restrictive for that workflow.
     20/min still protects Nominatim from abuse.

  4. CarryForwardPrompt execution step left as TODO in FRONTEND spec
     Rationale: BACKEND correction C1 runs in parallel with FRONTEND.
     FRONTEND can implement all other features without blocking on C1.
     Better to unblock FRONTEND immediately than wait for C1 to be complete.

  5. Bug classification framework adopted (SS-25–30)
     Rationale: Pattern of trivial-to-substantial reclassification observed
     across BACKEND and ARCHITECT threads. Framework addresses discovery,
     mandatory re-evaluation, reclassification escalation, and documentation
     requirements. Applies to all jobs from this date forward.

  6. SQLite plaintext at rest: ACCEPTED AS-IS (confirmed from BACKEND's decision)
     Rationale: Personal travel data, low sensitivity, OS-level FileVault mitigates.

  7. BRD header inconsistency: corrected v2.2 → v2.3
     Rationale: The BRD content reflects v2.3 (EX-01 is listed in changelog)
     but the header was never updated. Corrected for traceability.
