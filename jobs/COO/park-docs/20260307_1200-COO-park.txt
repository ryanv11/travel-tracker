DECISIONS MADE AND RATIONALE
================================================================================

1. DEPENDENCY CHAIN: ARCHITECT → DATABASE → BACKEND → FRONTEND → QA/DOCS → INTEGRATIONS
   Rationale: linear critical path is the only valid option — no job can
   meaningfully begin without the Architect's data model and tech blueprint.
   QA and DOCS can run in parallel once FRONTEND has stable features.
   INTEGRATIONS is explicitly post-MVP.

2. SHADING STATES ARE COMPUTED, NOT STORED
   Map shading state for any location is derived from trip data at query time.
   Defined "completed trip" as status in (Review Pending, Locked) — Review
   Pending trips have happened and must count toward visit state.

3. PLACE = CITY-LEVEL ASSOCIATION
   "Place" in the BRD is interpreted as a trip-place association at city level.
   Country and region shading roll up from city visits. Architect to confirm.

4. GEOCODING: QUEUE-AND-RESOLVE PATTERN (GE-12)
   Cities with no internet at entry time are created immediately (PENDING).
   Coordinates resolved silently in background when connectivity returns.
   User is never interrupted. Confirmed by project owner.

5. Q1 — DESKTOP PACKAGING: TARGET .APP, LOCALHOST FOR BETA
   Tech stack must support both modes with zero code change between them.
   Packaging is a delivery step, not an architectural decision. Architect to
   recommend Electron or Tauri.

6. Q2 — RATINGS: 1–5 STARS, RESTAURANTS AND HOTELS, SORT + FILTER REQUIRED
   Rating sort/filter applies at trip level and city level (cross-visit).
   Confirmed as a new functional requirement not in BRD v2.1. Added to v2.2.

7. Q3 — GEOGRAPHY: BUNDLED POLYGONS + NOMINATIM GEOCODING
   Country/region boundaries bundled (Natural Earth). Cities geocoded via
   Nominatim on first entry, stored permanently. Offline-tolerant via
   queue-and-resolve pattern (see decision 4).

8. Q4 — CARRY-FORWARD: AUTO-SUGGEST WITH CARRIED_FROM FLAG
   Prompt presented on new trip creation to a previously visited city.
   User confirms per item. Carried-forward items flagged with
   carried_from_item_id and is_carried_forward in data model.
