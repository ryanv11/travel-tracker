DECISIONS MADE
================================================================================

1. DEPENDENCY CHAIN CONFIRMED
   ARCHITECT → DATABASE → BACKEND → FRONTEND → QA/DOCS (parallel) → INTEGRATIONS
   Rationale: linear critical path; QA and DOCS can run in parallel once
   FRONTEND has stable features; INTEGRATIONS is post-MVP.

2. ARCHITECT IS THE ONLY ACTIVE JOB
   All other jobs remain blocked until Architect delivers the data model,
   tech blueprint, and mapping recommendation.
   Rationale: no DATABASE schema can be written, no BACKEND API can be designed,
   and no FRONTEND can be planned without the Architect's foundational decisions.

3. SHADING STATES ARE COMPUTED, NOT STORED
   The map shading state for any location is derived from trip data at query
   time. It is not stored as a field anywhere. This was codified into the
   Architect spec. The COO has defined the computation rules (Task 2 of spec)
   to give the Architect a clear starting point, but the Architect may refine
   them.

4. DEFINITION OF "COMPLETED TRIP" FOR SHADING PURPOSES
   Trips with status Review Pending count as "completed" for shading purposes
   (along with Locked). Rationale: a Review Pending trip has already happened;
   excluding it would cause a gap in map shading between trip end date and
   review completion.

5. "PLACE" IS A CITY-LEVEL ENTITY
   The COO has interpreted the BRD definition of "Place" as a city-level
   trip-place association. Country and region shading roll up from city-level
   visits. The Architect is asked to confirm or challenge this interpretation.

6. Q1 (DESKTOP PACKAGING) FLAGGED AS A POTENTIAL BLOCKER
   NF-01 ("local Mac desktop app") is ambiguous about whether a packaged .app
   or a browser-at-localhost is required. This has been included in the
   Architect spec as Task 5, Q1, and flagged as a potential blocker that may
   need project owner resolution before the Architect can finalise the
   technology stack.
