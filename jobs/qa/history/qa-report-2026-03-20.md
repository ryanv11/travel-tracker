## Part 2 — New Backend Bugs

### BUG-A: PATCH /api/trips/:id — date validation bypassed when only one date field is sent
**Severity:** Medium
**File:** `src/backend/validation/trips.schemas.ts` — `UpdateTripSchema` refine
**Reproduction:**
```bash
# Create trip with start_date=2026-06-01, end_date=2026-06-10
# Then PATCH with only end_date set before start_date:
curl -X PATCH .../api/trips/:id \
  -H "Content-Type: application/json" \
  -d '{"end_date": "2026-05-01"}'   # before existing start_date — accepted!
```
**Root cause:** The refine function is:
```ts
if (d.start_date && d.end_date) return d.end_date >= d.start_date;
return true;  // ← passes when only one date is present
```
When only `end_date` is sent, validation returns `true` regardless of whether the new end_date falls before the existing start_date. The backend has no cross-field validation against the stored record.

**Impact:** A trip can be left with `end_date < start_date` via partial PATCH.
**Fix direction:** Either (a) validate the end_date against the stored trip's start_date in the route handler, or (b) require `start_date` whenever `end_date` is provided.

---

### BUG-B: Admin list endpoints return camelCase keys; frontend expects snake_case
**Severity:** High (visual regression — all admin list items render as "Inactive")
**File:** `src/backend/routes/admin.ts` — `createAdminListRouter()` factory
**Root cause:**
```ts
res.json(rows);  // raw Drizzle rows: { isActive, createdAt, ... }
```
Drizzle's `.select().from(table)` returns camelCase column names. All admin list endpoints (categories, activities, companions) use this factory and return `isActive` instead of the `is_active` the frontend expects.

**Impact:** Frontend `AdminListPage` reads `item.is_active` — which is `undefined` for all rows — so all items display as "Inactive" in the admin UI.

**Fix direction:** Add explicit column serialization in the factory (same pattern as the countries endpoint which already does explicit `region_tier_enabled: c.regionTierEnabled` mapping).

---

### ~~BUG-C~~: region_tier_label retained when tier disabled — CLOSED (by design)
**Status:** Closed — intentional behaviour
**Rationale:** Retaining the label when `region_tier_enabled` is set to false acts as "memory": if the admin re-enables the tier, the label is automatically restored without re-entry. Clearing the label on disable was previously attempted and broke the re-enable flow (validation requires a label when tier is enabled, but the label had just been cleared).

**Nice-to-have (NTH-04):** The admin UI should grey out / disable the `region_tier_label` input while the tier is disabled, so it's clear the value is retained but inactive. No backend change needed — UI-only.

## Part 3 — Security Findings

### SEC-01: GET /api/cities/:id/items returns all users' completed items (no userId filter)
**Severity:** High (data exposure across users)
**File:** `src/backend/routes/cities.ts`
**Endpoint:** `GET /api/cities/:id/items`

The query selects items scoped to `cityId` but does NOT filter by `userId`:
```ts
.where(and(eq(items.tripId, ...), eq(items.status, 'visited')))
// Missing: eq(items.userId, userId)
```
Any authenticated user can retrieve completed items belonging to other users by querying a city ID.

**Fix direction:** Add `eq(items.userId, userId)` to the where clause. `userId = req.user!.id` is already available in the handler.

---

### SEC-02: POST carry-forward — source_item_ids are not ownership-checked
**Severity:** Medium (cross-user data copy)
**File:** `src/backend/routes/places.ts` — POST `/:placeId/carry-forward`

The handler verifies that source item IDs exist, but not that they belong to the current user:
```ts
const foundItems = await db
  .select({ id: items.id })
  .from(items)
  .where(inArray(items.id, sourceItemIds));
// Missing: and(eq(items.userId, userId))
```
User A can carry forward items owned by user B into their own trip.

**Fix direction:** Add `eq(items.userId, userId)` to the ownership check query.

---

### SEC-03: DELETE /api/trips/:tripId/places/:placeId/activities/:activityId — no ownership check
**Severity:** Medium (cross-user mutation)
**File:** `src/backend/routes/places.ts` — DELETE `/:placeId/activities/:activityId`

The handler deletes by `placeId` + `activityId` only, without verifying that the place belongs to a trip owned by the current user:
```ts
await db
  .delete(tripPlaceActivitiesMap)
  .where(and(
    eq(tripPlaceActivitiesMap.tripPlaceId, placeId),
    eq(tripPlaceActivitiesMap.activityId, activityId),
  ));
```
Any authenticated user can remove activity tags from any place on any trip.

**Fix direction:** Before deleting, verify the place belongs to a trip owned by `req.user!.id` (use `placeRepository.findById(userId, placeId)` which already does this join).

## Part 5 — Nice-to-Have Flags

### NTH-01: Bulk delete has no undo / recovery
**Justification:** `window.confirm` is a hard-to-miss modal, but once confirmed, deletion is permanent with no undo. Trips contain complex nested data (places, items, extensions). A toast with "Undo" (5-second window) before the actual API calls would provide a safety net proportional to the destructive action. Low priority for a personal app but worth noting.

### NTH-02: Carry-forward does not copy flight / car_rental extension data
**Justification:** Flights (airline, flight number, booking ref) and car rentals are silently dropped during carry-forward. Only restaurant name/cuisine and hotel property/dates are copied. If a user books the same flight route for a repeat trip, they'd expect those details to carry. A UI note ("Flight details are not carried forward") or extending the copy to include these fields would reduce confusion. Currently undocumented in UI.

### NTH-03: Status filter chips don't show per-status counts
**Justification:** The status chips (All / Planning / Active / Review / Locked) show no count. Adding `(n)` to each chip would let users immediately see distribution without clicking each filter. Low effort, high scan-ability improvement — consistent with the `My Trips (n)` count badge already present.
