AREAS THAT NEED CLOSEST SCRUTINY

These are the areas most likely to contain bugs based on complexity:

1. ITEM STATUS WORKFLOW (FLAG-F1 — see below)
   The API reference doc uses "booked" and "skipped" but the actual Zod schema uses
   "consider", "confirmed", "completed", "cancelled", "next_time". Please verify that
   the status values the frontend sends match what the backend accepts. The frontend
   follows the Zod schema.

2. MAP SHADING FEATURE-STATE
   Country shading uses MapLibre feature-state (not data-driven properties). Verify:
   - Countries with a shading colour display the correct colour
   - Countries with no shading are transparent (not black or white)
   - Clicking a country at zoom < 4 does not load region data
   - Clicking a country at zoom >= 4 triggers region shading load for that country

3. CARRY-FORWARD FLOW (multi-step, easy to break)
   Steps: add a place to a trip → API returns carry-forward candidates → modal appears
   → user selects items → POST carry-forward → success message → modal closes.
   Test: zero candidates (modal should not appear), partial selection, all deselected
   (button should be disabled).

4. TRIP STATUS TRANSITIONS
   Valid transitions only:
     planning → active
     active → review_pending
     review_pending → locked
     locked → active  (unlock)
   Verify that invalid transitions are not shown as buttons. Verify that the locked
   banner appears and edit controls are disabled when status = 'locked'.

5. POST-TRIP REVIEW → LOCK
   - Confirm dialog must appear before lock is called
   - After lock, page should navigate away (to /trips)
   - Rating + postVisitNotes should only be editable for rateable item types
     (restaurant, hotel, experience) when status = completed

6. ADMIN COUNTRY TOGGLE (250 rows)
   - Search filters in real time
   - Checkbox toggle calls PATCH; verify no double-fire on rapid clicks
   - region_tier_label badge appears only when region_tier_enabled = true

7. SEC-12 — URL SANITISATION
   If any trip has a photo_album_ref that is NOT an https:// URL, verify no link is
   rendered (the sanitiser should return null and suppress the anchor tag).

KNOWN OPEN FLAGS

FLAG-F1 — Item status terminology mismatch
  API reference doc says "booked"/"skipped"; Zod schema says "confirmed"/"cancelled".
  Frontend uses the schema. QA should verify backend actually accepts schema values
  and flag to COO if a correction is needed on either side.

FLAG-F2 — No standalone GET /api/cities list endpoint
  City pins are derived from trip places. If QA tests a scenario where a trip has
  no places, the map will show no pins — this is expected behaviour, not a bug.
