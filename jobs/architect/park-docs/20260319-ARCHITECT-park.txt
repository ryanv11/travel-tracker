================================================================================
DECISIONS MADE (ADL-16 through ADL-19)
================================================================================

ADL-16: user_id on trips + items + trip_places (indexed FK, TEXT UUID)
        — endorsed the proposed trips+items boundary
        — ADDED trip_places beyond the proposal (cross-trip query RLS rationale)
        — rejected composite PK; endorsed indexed FK per COO proposal

ADL-17: OAuth (Google) PKCE + JWT access/refresh tokens + jose library
        — rejected email+password (unnecessary complexity for personal-family app)
        — rejected session cookies (Electron and iOS platform incompatibility)
        — rejected Passport/Lucia/Auth.js (wrong fit or Next.js-specific)
        — mandated: access tokens in memory only, NOT localStorage

ADL-18: Repository layer for all user-scoped query patterns
        — rejected per-route explicit filtering (no enforcement mechanism)
        — rejected middleware injection (Drizzle has no interceptor API)
        — repository layer gives enforcement + testability + Postgres RLS readiness
        — identified 6 specific cross-user leakage risks in current routes

ADL-19: Schema anomaly — undocumented columns flagged, removal recommended
        — process violation: schema changed without Architect review
        — action: COO to investigate, DATABASE to remove via migration

BUGS DISCOVERED
================================================================================

[MAJOR] Schema drift: four undocumented columns in trips and trip_places
  — trips.ownerAccountId (text, nullable, no FK)
  — trips.subscriptionId (text, nullable, no FK)
  — trips.createdByAccountId (text, nullable, no FK)
  — trip_places.createdByAccountId (text, nullable, no FK)
  Disposition: escalated to COO; recommended removal via migration (ADL-19)
  Process: not corrupting existing data (nullable/unused); severity is MAJOR
  because it represents unapproved schema change and will create confusion
  during NR-14 implementation if not resolved first.

[MAJOR — not yet live, will become CRITICAL when multi-user enabled]
  Cross-user data leakage risk in existing route patterns:
  — trips.ts getTripOrThrow(): no user_id filter
  — items.ts GET/PATCH /:itemId: verifies tripId but not trip ownership
  — places.ts: tripId only, not user ownership
  — cities.ts carry-forward: no user scoping (leaks cross-user next_time items)
  — cities.ts city items: no user scoping (leaks cross-user completed items)
  — map.ts shading: aggregates across all users
  Disposition: escalated to COO via recommendations document; must be fixed
  via repository layer (ADL-18) before any multi-user access is enabled.
  Currently safe (single-user app). Will be CRITICAL the moment a second user
  can authenticate.
