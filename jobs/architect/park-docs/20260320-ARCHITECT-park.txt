SESSION: Clerk auth assessment — NR-14 direction revision

================================================================================
WHAT HAPPENED THIS SESSION
================================================================================

COO query received: PO has directed Clerk as managed auth platform. Asked Architect
to assess technical blockers and provide implementation guidance.

Previous recommendation (ADL-17) was jose + roll-your-own OAuth PKCE with Google
as initial provider. PO has rejected the "family app" framing — ambition is broader
distribution, managed auth is the right call at scale.

Assessment outcome: NO TECHNICAL BLOCKER. Clerk is compatible with this stack.

Four questions answered:

1. EXPRESS / REQ.USER COMPATIBILITY
   Clerk issues standard JWTs. Backend verifies via jose + Clerk's JWKS endpoint.
   req.user shape is unchanged. authenticate middleware signature is unchanged.
   Route handlers require no modification. Zero Express incompatibility.

2. ELECTRON / iOS CAPACITOR
   Electron: open Clerk hosted sign-in in system browser (shell.openExternal),
   receive callback on custom URL scheme (traveltracker://auth/callback). Same
   pattern as ADL-17 specified. No new complexity. Deferred to Phase 1 packaging.
   iOS Capacitor: Clerk React SDK works in WebView. Hosted backend unchanged.
   Deferred to Phase 3. No concern.

3. LOCK-IN RISK
   Moderate but managed by architectural seam rule: backend must never import
   @clerk/* packages. JWKS URI is an env variable. Migration = one line change + user
   export. If seam rule is violated, migration cost rises to ~1 week. Seam rule
   must be enforced at code review and gate review.
   Frontend: use Clerk hosted sign-in page, not embedded <SignIn /> components.
   Minimises replacement surface.

4. SCHEMA IMPLICATIONS
   users table: KEEP. Local table is required for FK integrity (trips, items,
   trip_places all reference users.id). Shape changes: DROP oauth_provider,
   oauth_subject; ADD clerk_id TEXT NOT NULL UNIQUE.
   refresh_tokens table: DROP (do not create). Clerk owns token lifecycle.
   All ADL-16 user_id FK decisions: unchanged.
   All ADL-18 repository layer decisions: unchanged.

================================================================================

  ADL-17 → SUPERSEDED by ADL-20
  ADL-20: Clerk managed auth. jose JWKS verification in backend. No Clerk SDK

          refresh_tokens not created.
