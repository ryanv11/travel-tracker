## Context

The app is currently single-owner. We are planning to expand access to non-owner
users — first deployment will be a trusted individual, but the architecture must
be specced for general multi-user access. We do not want to revisit the security
model when the second user is added.

NR-14 is a formal go/no-go gate that must pass before any non-owner accesses the
app. OP-06 is the hardening checklist that defines what passing NR-14 means. Neither
document exists yet — producing them is the primary deliverable of this brief.

---

## Access model

The app has two distinct classes of non-owner authenticated user that must not be
collapsed into a single role. Where the proposed controls are insufficient to
distinguish between them, introduce additional access states rather than simplifying:

| Role | Description |
|---|---|
| **Owner** | The account that created the app instance. Full access to all resources and admin functions. |
| **Explicitly shared** | An authenticated user who has been explicitly granted access to one or more trips as a companion. Access is scoped to the resources they have been granted. |
| **Authenticated-but-ungranted** | An authenticated user with a valid token who has no explicit grant to any resource. Must be treated as having no access to owner data — not as a trusted user awaiting permission. |

Unauthenticated requests are not a role — they are a system failure mode. Any request
without a valid token must be rejected at the authentication layer before reaching any
application logic. No access matrix column is needed; the checklist must simply verify
that no unauthenticated request can reach any resource, and specify the exact response
(status code, body, log behaviour) for this case.

The distinction between explicitly-shared and authenticated-but-ungranted is critical.
A valid Clerk token is not sufficient to access any resource — explicit grants are
required. The architecture must enforce this distinction at the data layer, not only
at the route layer.

## Deliverable 1: OP-06 Hardening Checklist

A structured checklist document covering the following domains. Each item must be
concrete enough to be closed objectively — if it cannot be verified by a specific
test, observable output, or reviewable artefact, it is not specific enough.

### Required content per domain

**1. Trust boundaries**
Named and explicit — not prose. What is inside the trust boundary, what is outside,
what crosses it and under what conditions. Diagram if it aids precision.

**2. Resource/action access matrix**
A table. Rows are resources (trips, places, items, cities, admin panel, map shading,
companions, categories, activities, trip metadata). Columns are the three authenticated
roles: owner / explicitly-shared / authenticated-but-ungranted.
Each cell: permitted / denied / scoped (describe scope) / not applicable.
No ambiguous cells — if it's unclear, flag it rather than leaving it blank.
Unauthenticated requests are handled before this matrix applies — see token validation.

**3. Token/claim validation**
Exact claims required from Clerk JWT. Exact validation steps in order. What happens
if any step fails — return code, log level, session behaviour. Pseudocode or decision
tree preferred over prose.

**4. Failure behaviour and response minimisation**
Explicit per failure mode. Not "return an error" — return what HTTP status, what
response body, what is logged, does the session terminate.

Cover the following failure modes:
- Missing token (no auth header)
- Invalid token (malformed or bad signature)
- Expired token
- Valid token but no grant exists for the requested resource (authenticated-but-ungranted)
- Valid token, grant exists, but resource does not exist

The first three are authentication failures — rejected before application logic.
The last two are authorisation failures — reached after a valid token is confirmed.
The checklist must verify these are handled at distinct layers with appropriate responses.

For inaccessible resources, specify the response strategy explicitly:
- Does a 404 leak resource existence to an authenticated-but-ungranted user?
- Do list endpoints return empty arrays or omit resources silently?
- Is count metadata (e.g. total trips) visible to users without access?
- Are resource names, dates, or any other metadata visible before access is confirmed?

The goal is response minimisation — no information about a resource should be
accessible to a user without an explicit grant, including its existence.

**5. Isolation enforcement layers**
Specify which combination of the following layers enforces user isolation, and what
each layer is responsible for. If a layer is not used, state why and confirm that
omitting it does not create a gap:

- **Route guards** — what is checked at the route level, and what is assumed to be
  handled downstream
- **Repository scoping** — how user_id is threaded through repository calls and
  whether it can be bypassed
- **Query predicates** — whether WHERE user_id = ? is present on every query that
  touches user-owned data, and how this is verified
- **Database constraints** — whether row-level constraints or foreign key relationships
  enforce ownership at the DB level, independent of application logic
- **Combination rationale** — which layers are defence-in-depth and which are primary
  controls; what happens if any single layer fails

This section must be specific enough to serve as a code review checklist — reviewers
must be able to verify that isolation is correctly implemented without relying on
trust in the implementing agent.

**6. Verification criteria**
Each checklist item must have a named verification method:
- Contract test (specify the test or describe what must be written)
- Manual UAT step (specific steps + expected outcome)
- Code review checkpoint (what the reviewer confirms)

A checklist item is not closeable without a verification criterion.

## Known issues to address

**Clerk JWKS firewall (Issue #23)**
`BYPASS_AUTH=true` is currently in `.env.local` because the devcontainer firewall
blocks Clerk's JWKS endpoint. Real auth must work before NR-14 can pass — this is
a hard dependency. The checklist should include a specific item for this, and the
assessment should flag it as FAIL until resolved.

**Data isolation / user_id scoping**
The app has `user_id` columns on trips, items, and trip_places, and repositories
are nominally user-scoped. However, isolation must be verified as consistent at
every layer. The isolation enforcement section above is the primary vehicle for this.
Cross-user data leakage is the highest-risk area.

**Admin panel**
Currently any authenticated user can perform any admin action. The access matrix
must explicitly address what each role can and cannot do in admin, and the
checklist must include verification that this is enforced.

**AD-07/08/09 — per-user admin split**
Per-user map shading (AD-07), per-user companions (AD-08), and owner-only
deactivation (AD-09) are not yet implemented. The checklist should flag whether
these are in or out of NR-14 scope — if out, document the accepted risk explicitly.

**Global seed data**
Countries, regions, categories, and activities are global. Confirm whether
read access to these by non-owner users is an accepted risk or requires control.
Specify this explicitly in the access matrix.

  and gaps
