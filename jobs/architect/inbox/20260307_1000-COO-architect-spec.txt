TASK 1 — DATA MODEL

Design the full entity-relationship model for the system. Provide a structured

--------------------------------------------------------------------------------
1a. GEOGRAPHIC HIERARCHY
--------------------------------------------------------------------------------

The app uses a three-tier geographic structure: Country → Region → City.
See BRD sections 5.2 (GE-01 through GE-09).

  - Countries table: country code, name, whether Region tier is enabled for
    this country, and what the Region tier is called (State / Province /
    Territory / etc.)
  - Regions table: links to parent country. Only populated for countries
    with the Region tier enabled.
  - Cities table: links to parent country (always) and parent region (where
    the Region tier is enabled). Must store latitude and longitude for map pin
    placement.

Design decisions to address:

  GEOGRAPHIC DATA SOURCE — this is a question that will need project owner
  input (see Task 5), but provide your recommended approach and design
  accordingly. Options:

    Option A — Fully bundled: ship with a static dataset of all countries,
      major regions, and key cities pre-loaded. No external service needed.
      City coordinates included in the bundle.

    Option B — User-entered: user creates city records on-the-fly when
      logging trips. No pre-populated data. User enters coordinates manually
      or the app geocodes on lookup.

    Option C — Hybrid (recommended starting point): core countries and
      regions are bundled (small, stable dataset). Cities are created on
      entry, with coordinates populated via a geocoding API call (requires
      internet at entry time, stored locally after that).

  Provide your recommendation and design for the option you choose. If you
  make an assumption, document it clearly.

  DEFAULT COUNTRY CONFIGURATION (GE-05, GE-06): the app ships knowing which
  countries have Region tiers and what they are called (US→States,
  Australia→States, Canada→Provinces, India→States, most of Europe→disabled).
  This implies either a bundled countries table or a seed data file. Design
  for whichever fits your geographic data source recommendation.

--------------------------------------------------------------------------------
1b. TRIPS
--------------------------------------------------------------------------------

Core entity. See BRD TR-01 through TR-10.

Required attributes:
  - Name, start date, end date
  - Status: Planning | Active | Review Pending | Locked
  - Photo album reference (text/URL — no file storage, reference only, PH-01)
  - created_at, updated_at

Required many-to-many relationships (each from user-managed admin lists):
  - Trip Categories (TR-03)
  - Activities at trip level (TR-04 — activities can also attach to a
    specific place within the trip; design both)
  - Companions (TR-02)

Locked behaviour (TR-06, TR-07):
  - Locked trips are read-only. The data model should support an explicit
    lock flag (or rely on the status field) — state your recommendation.

Multiple trips to the same place are separate dated records — there is no
deduplication at the trip level (TR-08).

--------------------------------------------------------------------------------
1c. PLACES (Trip-Place associations)
--------------------------------------------------------------------------------

Per the BRD definition, a "Place" is a country and/or city visited within a
trip. Multiple places can belong to one trip (TR-01: "one or more destinations").

Design requirements:
  - A trip has one or more places
  - Each place is associated with a city (which implies region and country
    via the geographic hierarchy — do not store country/region redundantly)
  - Activities can attach at place level as well as trip level (TR-04)
  - "Next time" item carry-forward (IT-07) operates at place level — when
    creating a new trip to the same city, "Next time" items from prior trips
    to that city can be carried forward

Key design question: should Place be a standalone entity (places table with
its own ID, referenced by trips via a junction table), or should the trip-place
junction table itself carry the place-level metadata? Design whichever you
recommend and explain why.

--------------------------------------------------------------------------------
1d. ITEMS
--------------------------------------------------------------------------------

Items are logged against a trip, or against a specific place within a trip.
See BRD IT-01 through IT-07 and the type-specific fields in BRD 5.6.

  Item types:   Restaurant, Hotel, Flight, Car Rental, Experience, Note
  Item status:  Consider | Confirmed | Completed | Cancelled | Next time

All items share:
  - trip_id (required)
  - trip_place_id (optional — item may be trip-level rather than place-level)
  - item_type
  - status
  - notes (free text — all types, IT-04)
  - created_at, updated_at

Type-specific structured fields (see BRD 5.6 for full field lists):

  Flight:      airline, flight_number, departure_airport, arrival_airport,
               departure_datetime, arrival_datetime, booking_reference, seat
               (FL-01, FL-02) — each flight leg is a separate item (FL-01)

  Hotel:       property_name, address, check_in_date, check_out_date,
               booking_reference, confirmation_number (HT-01)
               Duration is calculated, not stored (HT-02)

  Car rental:  provider, pickup_location, dropoff_location, pickup_datetime,
               dropoff_datetime, booking_reference, vehicle_class (CR-01)

  Restaurant:  name, neighbourhood_area, cuisine_type, source
               (RS-01)
               When Completed: rating, post_visit_notes (RS-03)

  Experience:  no structured fields beyond the base item
  Note:        no structured fields beyond the base item

Design decision required — how to model type-specific fields:

  Option A — Single wide table: one items table with all type-specific
    columns, nullable where not applicable to the current type.
    Pro: simple queries. Con: many nullable columns, hard to enforce type
    integrity, table gets wide as new types are added.

  Option B — Base + extension tables: items table holds shared fields;
    one extension table per type (item_flights, item_hotels, etc.) with a
    1:1 FK relationship to items.
    Pro: normalised, enforces required fields per type. Con: joins required.

  Option C — Base table + JSON column for type-specific attributes:
    items table holds shared fields plus a type_data column (TEXT in SQLite,
    JSONB in PostgreSQL). Type-specific fields stored as structured JSON.
    Pro: flexible, works identically in SQLite and PostgreSQL, easy migration.
    Con: cannot index or query individual type-specific fields efficiently
    without JSON path operators (PostgreSQL has these; SQLite is limited).

  Provide your recommendation with rationale. Consider the migration path
  (SQLite → PostgreSQL) as a factor.

Item carry-forward (IT-07): design whether items need a carried_from_item_id
reference column, or whether this is a pure application-layer operation with
no data model implication. State your decision.

--------------------------------------------------------------------------------

The following reference lists are user-managed via the admin panel (AD-01
through AD-06). All lists support add, edit, rename, and deactivate (not
hard delete — deactivated items are hidden from entry forms but preserved
on existing records).

  - trip_categories (id, name, is_active)
  - activities (id, name, is_active)

Design the soft-delete pattern consistently across all admin lists.

TASK 2 — MAP SHADING STATES AND COMPUTATION LOGIC
================================================================================

The BRD defines 7 shading states (including Never Visited as the default).
See BRD sections 5.3 and 5.4.

The shading state for any country, region, or city is COMPUTED from trip
data — it is not stored as a field. Define the computation logic that BACKEND
will implement, including the priority order and the definitions used.

COMPUTATION RULES (implement in this priority order):

  1. Active — any trip with status=Active includes this location
     → overrides all other states (MP-06)

  2. Visited multiple + planning — trips to this location include:
     2+ "completed" trips AND 1+ "planning" trips

  3. Visited multiple — 2+ "completed" trips, 0 "planning" trips

  4. Visited once + planning — exactly 1 "completed" trip AND 1+ "planning"
     trips

  5. Visited once — exactly 1 "completed" trip, 0 "planning" trips

  6. Planned — 0 "completed" trips, 1+ "planning" trips

  7. Never visited — 0 completed, 0 planning, 0 active (no trips logged)
     → this is the default; no shading is rendered

DEFINITIONS:
  "completed trip" = a trip with status in (Review Pending, Locked)
     Rationale: a trip in "Review Pending" has already happened — the user
     just hasn't finalised the record yet. It should count as visited.
  "planning trip"  = a trip with status = Planning
  "active trip"    = a trip with status = Active

ROLLUP LOGIC (country and region shading):
  - Country shading is computed by rolling up from all cities within that
    country that have been visited
  - Region shading is computed by rolling up from all cities within that
    region that have been visited
  - A country or region is "visited" if any city within it meets the
    visited threshold
  - The shading state at country/region level should reflect the aggregate
    state of all cities within it (e.g. a country is "Visited multiple times"
    if any city within it has 2+ completed trips)

Specify the database queries BACKEND will need to compute shading states
efficiently at country, region, and city level. Flag any recommended indexes.

Note: shading colours are stored in the map_shading_config table. The
computation logic returns a state_key (e.g. "visited_once"), which BACKEND
maps to the configured hex colour for rendering.

TASK 4 — MAPPING LIBRARY RECOMMENDATION
================================================================================

The map view requires (BRD sections 5.3, 5.4):
  - World map with country-level polygon shading (choropleth) — 7 states
  - Zoom to reveal region-level shading (where enabled) and city-level pins
  - Click on country, region, or city to view associated trip entries (MP-03)
  - Shading colours configurable per state, applied at render time from config
  - Internet required for tile rendering (NF-02); app logic and data are local
  - Must run in a browser (local or hosted — same frontend code)

This is Open Question OQ-01 from the BRD. Your recommendation will be treated
as the architectural decision.

Evaluate:

  Leaflet.js + OpenStreetMap
    Free, open source, mature, large community. Raster tiles. Good GeoJSON
    support for polygon shading. Simpler API. Raster shading transitions at
    zoom levels are less smooth.

  MapLibre GL JS + OpenStreetMap vector tiles
    Free, open source fork of Mapbox GL JS. Vector tiles. WebGL rendering.
    Smooth zoom through country → region → city. Better choropleth rendering.
    More powerful styling. Requires a vector tile provider (MapTiler free tier
    in development; self-hosted or low-cost CDN for production).

  Google Maps JavaScript API
    Polished and familiar. Requires API key. Has usage costs at scale.
    Choropleth shading requires custom GeoJSON overlay work (not built-in).
    Less suitable for open-source/low-cost personal project.

  Other options worth raising (Mapbox GL JS, D3.js for map only, etc.)

For each option you evaluate, state:
  - Tile provider needed and cost implications
  - GeoJSON boundary data source for country/region polygons
  - Suitability for choropleth shading at country and region level
  - Suitability for city-level pin markers with click events
  - Compatibility with your recommended frontend framework

Make a clear recommendation and justify it. Note any dependencies your
recommendation creates for the DATABASE or BACKEND layer.
