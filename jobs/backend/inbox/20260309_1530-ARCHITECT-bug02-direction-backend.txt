RE:   BUG-02 Direction — Return iso_3166_2 in region shading responses

---

OVERVIEW

Region shading is broken because RegionLayer.tsx applies feature-state using
integer region_id values (e.g. 1) but the GeoJSON uses ISO 3166-2 codes
(e.g. 'US-CA') as feature IDs via promoteId="iso_3166_2". They never match.

Database is adding an iso_3166_2 column to the regions table. Backend must
return this column in all region shading API responses.

DEPENDENCY: Wait for Database to confirm the migration is applied and the
column exists before merging backend changes. You can develop in parallel but
do not push until the schema change is in place.

---

CHANGE 1 — shading.service.ts: include iso_3166_2 in region queries

File: src/backend/services/shading.service.ts

The RegionShadingResult interface and getRegionShading() function currently
return region_id and region_name. Add iso3166_2:

  export interface RegionShadingResult extends ShadingResult {
    regionId: number;
    regionName: string;
    iso3166_2: string;          // ← ADD THIS
  }

In getRegionShading(), the .select() call must include:
  iso3166_2: regions.iso3166_2,   // ← ADD TO SELECT

And in the .map() at the end:
  return {
    regionId: r.regionId,
    regionName: r.regionName,
    iso3166_2: r.iso3166_2,       // ← ADD TO RETURN
    ...buildResult(stateKey, config),
  };
