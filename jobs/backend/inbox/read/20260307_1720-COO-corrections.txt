CORRECTION 1 — FIRST-LAUNCH SEED: STARTUP MUST INCLUDE ADMIN DATA
================================================================================

PROBLEM:
  The original BACKEND spec defines this startup sequence:
    1. getDb()
    2. startupService.seedCountries()
    3. geocodingService.processQueue()

  This is incomplete. trip_categories, activities, companions, and
  map_shading_config are seeded by `npm run db:seed` — a manual script.
  If a user starts the app without first running that script, these tables
  will be empty. The app would launch with no trip categories, no companions,
  no shading config — unusable on first launch.

CORRECTION:
  The startup service must also seed admin data if the tables are empty.
  Import and call the same seed functions used by the DATABASE seed script,
  or call the script directly. Do NOT require the user to run `db:seed` manually.

  Updated startup sequence:
    1. getDb()                                    — verify DB connection
    2. startupService.seedAdminData()             — seeds trip_categories,
                                                    activities, companions,
                                                    map_shading_config
                                                    (idempotent — safe to call
                                                    every startup via
                                                    onConflictDoNothing)
    3. startupService.seedCountries()             — seeds countries if empty
    4. startupService.seedRegions()               — seeds regions if empty
                                                    (see Correction 2)
    5. geocodingService.processQueue()            — resolve pending cities
    6. schedule geocoding every 15 minutes

  The admin seed data is already implemented in src/backend/db/seed.ts.
  Import and call those functions directly — do not duplicate the data.

  Log each seeding step at info level (e.g. "Admin data: already seeded, skipping").

CORRECTION 2 — REGIONS NOT SEEDED + REGION_ID IS OPTIONAL (NOT REQUIRED)
================================================================================

PROBLEM:
  The original BACKEND spec states: "if country.region_tier_enabled = 1,
  region_id must be provided [when creating a city]."

  This is wrong for two reasons:
    a) No region data is pre-loaded for region-tier countries. A user would
       be unable to create any city in the US, Australia, or Canada without
       first manually creating all 50 states / 8 states / 13 provinces
       via the admin panel. That is terrible UX.
    b) The BRD says "up to three levels" (GE-01). "Up to" means optional.

CORRECTION A — RELAX THE REGION_ID REQUIREMENT:
  region_id is OPTIONAL on city creation even when region_tier_enabled = 1.
  Enforce only the inverse rule: if country.region_tier_enabled = 0,
  region_id MUST be NULL (reject with 400 if supplied).

  A city without a region_id in a region-tier country:
    - Is valid and fully usable for trip and item logging
    - Appears in country-level shading queries
    - Does NOT appear in region-level shading (no region to roll up to)
    - Can have a region_id set later via PATCH if the user adds one

CORRECTION B — SEED REGIONS FOR US, AU, CA ON FIRST LAUNCH:
  The startup service must seed regions for the three countries where
  pre-populating regions is essential for usable UX. All others can be
  added by the user via the admin panel.

  Create: data/regions.json (same pattern as data/countries.json)

  File format:
    [
      { "country_code": "US", "name": "Alabama" },
      { "country_code": "US", "name": "Alaska" },
      ...
    ]

  startupService.seedRegions() implementation:
    - Count rows in regions table.
    - If > 0: log "Regions already seeded, skipping" and return.
    - If 0: load data/regions.json and insert all rows.
    - Use onConflictDoNothing() (no unique constraint on name+country, so
      just check count > 0 before inserting).
    - Log "✓ Regions seeded ({count} rows)"

  AUSTRALIA (8 entries):

  For all other region-tier countries (DE, IN, JP, BR, etc.):

CORRECTION 3 — CI PIPELINE (SEC-13): BACKEND OWNS SETUP
================================================================================

  The security spec (SEC-13) requires secret scanning, dependency scanning,
  and SAST to be running in CI before Phase 1 beta. BACKEND is responsible
  for configuring the CI pipeline.

  Set up GitHub Actions (or equivalent) with these steps:
    1. npm audit --audit-level=high
    2. Gitleaks secret scanning
    3. Semgrep SAST (p/typescript, p/react, p/nodejs, p/owasp-top-ten)

  Create: .github/workflows/security.yml

  This must be done before closing your thread. The CI pipeline running
  green (with any false positives suppressed and documented) is an
  acceptance criterion. Add it to your completion report.
