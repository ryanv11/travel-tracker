================================================================================
DECISIONS MADE
================================================================================

1. Path param validation response code
   Decision: return 400 (not 404) when id is non-numeric/zero/negative.
   Rationale: spec says "return 400 if invalid". The route returns
   { error: "Trip not found" } as the body — matches spec wording exactly
   (spec says the 404 body is { "error": "Trip not found" }; 400 body is
   not specified but using the same message is consistent and harmless).

2. Test infrastructure: server-test-app.ts
   Decision: created src/backend/server-test-app.ts as a dedicated test
   export of the Express app without the listen() call and startup tasks.
   Rationale: server.ts calls startup() which runs seeding and geocoding
   and calls app.listen(). Supertest needs the bare app object. Duplicating
   middleware config in a separate file is the correct approach — it keeps
   server.ts unchanged and does not introduce a test-conditional code path
   in the production server.

3. In-memory SQLite test database
   Decision: create a fresh :memory: libSQL client per test in beforeEach.
   Rationale: full isolation, no test file leakage, avoids relying on a
   real dev.db file being present. Applied the minimal DDL that matches
   the actual schema (including columns added in migration 0001:
   owner_account_id, subscription_id, created_by_account_id on trips;
   created_by_account_id on trip_places; iso_3166_2 on regions;
   subscription_id on map_shading_config).

4. Test categorisation: 7 schema unit tests + 7 route integration tests = 14
   Schema tests: pure Zod schema validation, no DB.
   Integration tests: supertest + in-memory SQLite via vi.mock on getDb().
