BUG-03: MAJOR — Carry-forward modal race condition [IMPLEMENT NOW]

Location: AddPlaceFlow component
Problem:  useEffect fires before the carry-forward candidates API query
          resolves. It sees an empty candidates array and calls onClose()
          immediately. The modal is never shown.

Fix:      Before acting on an empty candidates list, check that the query
          has actually completed (isFetched === true or isLoading === false).
          Only call onClose() when isFetched is true AND candidates is empty.
          While isFetching or isLoading, do nothing — wait for the query.

Acceptance: The carry-forward modal renders correctly when candidates exist.
            onClose() is only called after the query has settled and candidates
            are confirmed empty.

---

BUG-04: MAJOR — No UI path to revert review_pending → planning [IMPLEMENT NOW]

Location: ReviewPanel component (rendered by TripDetailPage for review_pending trips)
Problem:  ReviewPanel has no "Return to Planning" button. The backend supports
          the review_pending → planning transition, but there is no way to
          trigger it from the UI. AC-08 and AC-13 are partially failing.

Fix:      Add a "Return to Planning" button to ReviewPanel.
          - Button must show a confirmation dialog before acting
            (e.g. "Return this trip to planning? The review will be cleared.")
          - On confirm: call PATCH /api/trips/:id with { status: "planning" }
          - On success: invalidate the trip query and navigate/refresh

Acceptance: A review_pending trip can be reverted to planning status via the UI.
            The transition requires explicit user confirmation.

BUG-05: MINOR — Mark all Completed incorrectly includes next_time items [IMPLEMENT NOW]

Location: ReviewPanel — handleMarkAllCompleted
Problem:  The NON_CANCELLED array used for bulk completion includes 'next_time'
          items. next_time items should not be bulk-completed.

Fix:      Remove 'next_time' from the NON_CANCELLED array (or equivalent
          filter logic). Only items that are neither cancelled nor next_time
          should be included in the bulk complete action.

Acceptance: Bulk "Mark all Completed" does not change the status of next_time items.
