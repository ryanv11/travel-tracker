## Task 1 — Bulk delete undo window (Closes #33)

**File:** `src/frontend/components/TripList/TripsLayout.tsx`

**What to build:** After the user confirms bulk delete, instead of immediately
executing API calls, show a 5-second undo bar inline in the left panel. If the
user clicks Undo, cancel. If the window expires, proceed with deletions.

**Implementation guide:**

Add state to track a pending deletion:
```ts
const [pendingDelete, setPendingDelete] = useState<{ ids: Set<number>; timer: ReturnType<typeof setTimeout> } | null>(null);
```

In `handleBulkDelete`, after `window.confirm`:
1. Capture the current `selectedIds` snapshot into a local `Set<number>`
2. Set a 5-second `setTimeout` that executes the actual API loop (same logic as today)
3. Store `{ ids, timer }` in `pendingDelete` state
4. Call `exitSelectionMode()` immediately (the undo bar is outside selection mode)

Render the undo bar conditionally in the left panel header area (above the trip
list, below the search row) while `pendingDelete` is non-null:
```tsx
{pendingDelete && (
  <div className="mx-4 mb-2 px-3 py-2 bg-red-50 border border-red-200 rounded-md flex items-center justify-between text-sm">
    <span className="text-red-700">Deleting {pendingDelete.ids.size} trip{pendingDelete.ids.size === 1 ? '' : 's'}…</span>
    <button
      type="button"
      onClick={handleUndoDelete}
      className="text-red-600 font-medium hover:text-red-800"
    >
      Undo
    </button>
  </div>
)}
```

`handleUndoDelete`:
- `clearTimeout(pendingDelete.timer)`
- `setPendingDelete(null)`

The actual delete execution (moved into the timer callback):
- Same `for...of` loop calling `deleteTrip.mutateAsync`
- Same navigate-away logic for current trip
- Same `alert(...)` on partial failure
- Call `setPendingDelete(null)` in finally

**Existing `window.confirm` stays in place** — it remains the primary gate. The
undo window is a secondary safety net.

## Task 2 — Status filter chip counts (Closes #34)

**File:** `src/frontend/components/TripList/TripsLayout.tsx`

**What to build:** Show `(n)` count on each status chip.

**Implementation guide:**

Add a second `useTrips()` call with no filters, purely for counting:
```ts
const { data: allTrips = [] } = useTrips();  // no filters — for counts only
```

React Query will cache this independently from the filtered `trips` query, and it
will update reactively when trips are mutated.

Derive counts with `useMemo`:
```ts
const statusCounts = useMemo(() => {
  const counts: Record<string, number> = {};
  for (const t of allTrips) {
    counts[t.status] = (counts[t.status] ?? 0) + 1;
  }
  return counts;
}, [allTrips]);
```

Update the `STATUS_CHIPS` render to include the count:
```tsx
{chip.label}{chip.value ? ` (${statusCounts[chip.value] ?? 0})` : ''}
```

The "All" chip (`chip.value === ''`) shows no count — keep it as "All".
